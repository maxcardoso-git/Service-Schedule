  <files>frontend/src/pages/Receptionist.jsx</files>
  <action>
Create `frontend/src/pages/Receptionist.jsx` with three sections in a single-page layout:

**1. Today's Agenda (top, full width)**
- Page title: "Receptionist" with current date (e.g., "Friday, 14/03/2026") and a "New Booking" button (opens ReceptionistBookingDialog)
- Fetch today's bookings: `apiFetch('/admin/bookings')` via useQuery with queryKey `['receptionist-bookings']`. Filter client-side to today's date only (compare booking.startTime date to today).
- Fetch professionals list: `apiFetch('/admin/professionals')` via useQuery with queryKey `['professionals']`.
- Group bookings by professionalId. Display as columns (one per active professional) in a CSS grid (`grid-cols-1 md:grid-cols-2 lg:grid-cols-3 xl:grid-cols-4`).
- Each column: professional name header, then booking cards sorted by startTime ascending.
- Each booking card: time range (HH:mm format, pt-BR locale), client name, service name, StatusBadge (reuse STATUS_COLORS/STATUS_LABELS pattern from Calendar.jsx -- copy the StatusBadge component and constants into this file, do NOT import from Calendar.jsx since it's a page not a shared component).
- Clicking a booking card opens a simple detail view (inline or small dialog) showing status, client, service, time, and status transition buttons (same STATUS_TRANSITIONS pattern from Calendar.jsx). Use useMutation for PATCH `/admin/bookings/:id/status`.
- If no bookings today, show "No bookings scheduled for today" message.
- Auto-refresh: set refetchInterval: 30000 on the bookings query.

**2. Client Search (bottom-left card)**
- Card with title "Client Lookup".
- Phone input + Search button. On search: `apiFetch('/admin/clients/by-phone/' + encodeURIComponent(phone))`.
- If found: show client name, phone, and last visit date. To get last visit: `apiFetch('/admin/clients/' + client.id + '/appointments')` then take the most recent booking's startTime. Display as "Last visit: DD/MM/YYYY" or "No previous visits".
- If 404: show "Client not found" message.
- Loading and error states.

**3. Availability Check (bottom-right card)**
- Card with title "Check Availability".
- Three inputs: service dropdown (from services query, active only), date input (default today), and professional dropdown (optional, filtered by selected service using same logic as NewBookingDialog's filteredProfessionals).
- Fetch services: `apiFetch('/admin/services')` via useQuery, filter active.
- Fetch professionals: `apiFetch('/admin/professionals')` via useQuery (reuse same queryKey as agenda section).
- "Check" button: calls `apiFetch('/admin/bookings/availability', { method: 'POST', body: { serviceId, date, professionalId } })`.
- Show results: list of available time slots (HH:mm format) or reason message (NOT_WORKING, FULLY_BOOKED). If no professional selected, show a note "Select a professional to check slots".
- staleTime: 0 on availability query (always fresh).

**Layout:** Use a 2-row layout. Top row: agenda (full width). Bottom row: 2-column grid (`grid-cols-1 md:grid-cols-2 gap-6`) for client search and availability check.

**Styling:** Follow existing page patterns -- rounded-xl border bg-card p-4 for card containers, same font sizes, Tailwind classes as Calendar.jsx and Clients.jsx.

**State management:** useState for all local state (phone input, selected service/professional/date, search results, availability results). useQuery for data fetching. useMutation for status transitions.

Import Button, Input, Label from @/components/ui/*. Import toast from sonner. Import apiFetch and ApiError from @/lib/api.

**ReceptionistBookingDialog.jsx:**

Create a simplified 3-step booking dialog adapted from NewBookingDialog.jsx. Same Dialog/DialogContent/DialogHeader pattern.

Props: `{ open, onOpenChange, onBookingCreated }`

**Step 1: Phone Lookup (same as NewBookingDialog step 1)**
- Phone input + Search button.
- `apiFetch('/admin/clients/by-phone/' + phone)` on search.
- If found: show name + phone, "Select" button sets selectedClient and advances to step 2.
- If 404: inline registration form (name, phone, email optional). On submit: `apiFetch('/admin/clients', { method: 'POST', body })`. On success: set selectedClient, advance to step 2.
- Copy the logic directly from NewBookingDialog steps 1 (handlePhoneLookup, handleRegisterInline).

**Step 2: Service + Professional + Date + Slot (combines NewBookingDialog steps 2-4)**
- Show service list (from useQuery `/admin/services`, active only). Clicking a service selects it.
- Once service selected, show professional list (filtered by service, same filteredProfessionals logic from NewBookingDialog).
- Once professional selected, show date input (default today) + slot grid (availability query, same as NewBookingDialog step 4 SlotGrid pattern).
- All on one scrollable view, not sub-steps. Use sections with headers: "Service", "Professional", "Time Slot".
- Selected service/professional shown as highlighted cards (bg-primary text-primary-foreground). Allow re-selection by clicking a different option.
- "Next" button enabled only when all four are selected (service, professional, date, slot).

**Step 3: Confirm (same as NewBookingDialog step 5)**
- Summary showing client, service, professional, date, time (same SummaryRow pattern).
- "Confirm Booking" button: POST `/admin/bookings` (pre-reserve) then PATCH `/:id/status` CONFIRMED (same handleConfirm logic from NewBookingDialog).
- On success: toast, call onBookingCreated(), close dialog.

Step indicator: "Step N of 3" with labels: "Client", "Service & Slot", "Confirm".
Back button on steps 2 and 3.
Reset all state when dialog closes (useEffect on open===false).

  <files>prisma/schema.prisma, src/services/conversationService.js</files>
  <action>
1. Add ConversationLink model to prisma/schema.prisma (AFTER the Payment model, BEFORE AdminUser):
   ```prisma
   model ConversationLink {
     id             String   @id @default(uuid()) @db.Uuid
     bookingId      String   @unique @db.Uuid
     conversationId String   @db.VarChar(255)
     createdAt      DateTime @default(now()) @db.Timestamptz

     booking Booking @relation(fields: [bookingId], references: [id], onDelete: Cascade)

     @@index([conversationId])
     @@map("conversation_links")
   }
   ```

2. Add the inverse relation on the Booking model — add `conversationLink ConversationLink?` after the `payment` field.

3. Run `npx prisma migrate dev --name add_conversation_link` to create the migration.

4. Create `src/services/conversationService.js` with two exported functions:

   - `createConversationLink(bookingId, conversationId)`: Creates a ConversationLink row. This function does NOT catch errors — the caller (bookingService) wraps it in try/catch for fire-and-forget behavior.

   - `getBookingsByConversationId(conversationId)`: Queries `prisma.conversationLink.findMany({ where: { conversationId }, include: { booking: { include: { client: true, services: { include: { service: true } }, professional: true } } } })`. Returns `links.map(l => l.booking)` — the caller gets an array of booking objects, not ConversationLink wrappers.

   Import prisma from `../lib/prisma.js` (same pattern as bookingService.js).

1. In `src/services/bookingService.js`:
   - Import `createConversationLink` from `./conversationService.js`
   - Modify `createPreReservation` to accept `conversationId` in the destructured params: `{ clientId, professionalId, serviceId, startTime, idempotencyKey, conversationId }`
   - AFTER the booking is successfully created (after the `prisma.$transaction` block, before `return booking`), add fire-and-forget conversation link creation:
     ```js
     if (conversationId) {
       try {
         await createConversationLink(booking.id, conversationId);
       } catch (err) {
         console.warn('Failed to create conversation link', { bookingId: booking.id, error: err.message });
       }
     }
     ```
   - IMPORTANT: This block must be OUTSIDE the transaction — after `const booking = await prisma.$transaction(...)` succeeds. Do NOT put it inside the transaction callback.

2. In `src/routes/bookings.js`:
   - Import `getBookingsByConversationId` from `../services/conversationService.js`
   - Add `conversationId: z.string().max(255).optional()` to the POST `/` Zod body schema
   - Add a new `GET /` route BEFORE the existing POST routes (Express matches in order). Use the established validate + asyncHandler pattern:
     ```js
     router.get('/',
       validate({
         query: z.object({
           conversationId: z.string().min(1),
         }),
       }),
       asyncHandler(async (req, res) => {
         const bookings = await getBookingsByConversationId(req.query.conversationId);
         res.json({ data: bookings });
       })
     );
     ```
     Note: The Zod schema requires conversationId (not optional) — if absent, validation returns 400 automatically. This matches CONV-02: the endpoint requires a conversationId filter to prevent unbounded queries.

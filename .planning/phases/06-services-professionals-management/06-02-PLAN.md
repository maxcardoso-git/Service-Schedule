  Create `frontend/src/pages/admin/Professionals.jsx` following Users.jsx pattern, with added service assignment capability.

  **Page structure:**
  - Header: "Professionals" title + "New Professional" button (Plus icon)
  - Table columns: Name, Contact (email/phone), Services (badges showing assigned service names), Status, Actions
  - Actions: Switch for active toggle + Pencil edit button + a "Services" button (Scissors icon) to open service assignment dialog
  - StatusBadge, SkeletonRow, empty state, error banner (same as Users.jsx)
  - Services column: render `professional.services.map(ps => ps.service.name)` as comma-separated Badge components. Show "None" in muted text if empty.

  **CreateProfessionalDialog (controlled):**
  - Fields: name (required), email (optional), phone (optional)
  - Mutation: POST to `/admin/professionals` with `{ name, email: email || undefined, phone: phone || undefined }`
  - On success: invalidateQueries(['professionals']), toast.success, close, reset

  **EditProfessionalDialog (controlled, receives `professional` prop):**
  - Fields: name, email, phone (pre-populated)
  - Mutation: PUT to `/admin/professionals/${professional.id}` with updated fields
  - On success: invalidateQueries(['professionals']), toast.success, close

  **ServiceAssignmentDialog (controlled, receives `professional` prop):**
  - This is a dialog that shows all available services as a checklist
  - Fetch all services: `useQuery({ queryKey: ['services'], queryFn: () => apiFetch('/admin/services') })` -- NOTE: this will work even if Plan 01 runs after Plan 02 because Plan 02 should add its own GET endpoint. BUT since both plans are wave 1 and this query may fail if services endpoint isn't ready, handle the error gracefully.
  - ALTERNATIVE if `/admin/services` doesn't exist yet: use the service data already embedded in the professional's `services` array for currently assigned, and fetch from `/admin/services` for the full list. If the fetch fails, show an error message "Load services first".
  - Display each service as a row with: service name + a Switch/checkbox toggle
  - Currently assigned services (from `professional.services`) are toggled ON
  - Toggling ON: POST to `/admin/professionals/${professional.id}/services` with `{ serviceId }`
  - Toggling OFF: DELETE to `/admin/professionals/${professional.id}/services/${serviceId}`
  - Each toggle is an independent mutation (no bulk save). On success: invalidateQueries(['professionals']), toast.success
  - Use individual useMutation calls (not a shared one) since each toggle is independent. Or use a single mutation with dynamic args.

  **Main component:**
  - `useQuery({ queryKey: ['professionals'], queryFn: () => apiFetch('/admin/professionals') })`
  - `const professionals = data?.data ?? []`
  - toggleActiveMutation: PUT to `/admin/professionals/${id}` with `{ active: !currentActive }`
  - State: createOpen, editProfessional, assignProfessional (for each dialog)

  **Wire into router (App.jsx):**
  - Import Professionals from `@/pages/admin/Professionals`
  - Add route `{ path: '/admin/professionals', element: <Professionals /> }` inside ADMIN-only ProtectedRoute children
